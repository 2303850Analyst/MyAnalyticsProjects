# Instacart Reorder Prediction

## Описание проекта

Проект посвящён прогнозированию повторных покупок пользователей на основе истории заказов Instacart.

Цель проекта — построить модель, которая предсказывает, какие товары пользователь с высокой вероятностью купит снова в следующем заказе. Такой подход применим в рекомендательных системах, персонализации корзины и механиках повторных продаж.

Проект ориентирован на задачи продуктовой аналитики и аналитики-разработки: анализ поведения пользователей, feature engineering, построение baseline, обучение моделей и оценка качества рекомендаций через ranking-метрики.

---

## Бизнес-задача

В e-commerce и grocery-сервисах значительная часть заказов состоит из повторных покупок. Если заранее предсказать товары, которые пользователь готов купить снова, продукт может:

*   персонализировать рекомендации
*   ускорить сбор корзины
*   повысить конверсию в заказ
*   увеличить повторные покупки
*   улучшить пользовательский опыт

Практический сценарий:

```text
Пользователь открывает новый заказ → система показывает топ товаров, которые он, вероятно, купит снова
```

## Постановка ML-задачи

Задача формулируется как бинарная классификация на уровне пары `user_id` + `product_id`.

Целевая переменная:

```text
reordered = 1, если пользователь купил товар в следующем заказе
reordered = 0, если не купил
```

Признаки строятся только на исторических данных `prior`, а целевая переменная берётся из `train`. Это снижает риск утечки данных.

### 3. Feature Engineering

Признаки построены на уровне `user-product`.

Использованные признаки:

| Признак | Описание |

## Модели

В проекте сравниваются три подхода:

1. Baseline по `product_reorder_rate`
2. Logistic Regression
3. LightGBM

### Baseline

Правило:

```text
если product_reorder_rate > 0.5 → predict reordered = 1
```

Результат:

| Metric | Value |
|---|---:|
| Precision | 0.126 |
| Recall | 0.825 |
| F1 | 0.219 |

Baseline имеет высокий recall, но низкую precision. Это означает, что он находит много повторных покупок, но слишком часто ошибается.

---

### Logistic Regression

| Metric | Value |
|---|---:|
| ROC-AUC | 0.814 |
| Precision | 0.604 |
| Recall | 0.077 |
| F1 | 0.136 |

Логистическая регрессия хорошо разделяет классы по ROC-AUC, но при стандартном пороге 0.5 имеет низкий recall.

---

### LightGBM

| Metric | Value |
|---|---:|
| ROC-AUC | 0.829 |
| Precision | 0.619 |
| Recall | 0.162 |
| F1 | 0.257 |

LightGBM показывает лучший результат среди моделей. Модель лучше улавливает нелинейные зависимости между пользовательской историей, популярностью товара и давностью последней покупки.

---

## Сравнение моделей

| Model | ROC-AUC | Precision | Recall | F1 |
|---|---:|---:|---:|---:|
| Baseline | - | 0.126 | 0.825 | 0.219 |
| Logistic Regression | 0.814 | 0.604 | 0.077 | 0.136 |
| LightGBM | 0.829 | 0.619 | 0.162 | 0.257 |

Вывод:

*   baseline полезен как точка отсчёта
*   Logistic Regression улучшает precision, но теряет recall
*   LightGBM показывает лучший баланс и лучший F1
*   для рекомендательной задачи важнее не только бинарная классификация, но и качество ранжирования

## Ranking-метрики

Для рекомендательной задачи дополнительно рассчитаны `Precision@K` и `Recall@K`.

|---|---:|---:|
| Baseline | 0.165 | 0.835 |

Вывод:

*   модель лучше baseline по Precision@K и Recall@K
*   top-K метрики лучше отражают продуктовый сценарий, чем accuracy

## Feature Importance

Наиболее важные признаки для LightGBM:

*   `orders_since_last_purchase`
*   `user_product_orders`
*   `product_reorder_rate`
*   `user_reorder_rate`

Интерпретация:

*   чем чаще пользователь покупал товар, тем выше вероятность повторного заказа
*   чем меньше времени прошло с последней покупки, тем выше вероятность reorder
*   товары с высоким reorder rate чаще подходят для персонализированных рекомендаций

## Product Recommendations

На основе результатов проекта можно предложить следующие продуктовые решения:

*   показывать пользователю top-K товаров, которые он с высокой вероятностью купит снова
*   использовать медианный интервал между заказами для выбора момента коммуникации
*   отдельно работать с low-сегментом, где поведение менее предсказуемо
*   усиливать персонализацию для high-сегмента, где reorder rate максимален
*   использовать модель как основу для блока “Купить снова” или “Добавить в корзину повторно”

---

## Ограничения проекта

*   модель предсказывает только повторные покупки товаров, которые пользователь уже покупал
*   cold-start для новых пользователей и новых товаров не решается
*   стандартный порог 0.5 не оптимизирован под бизнес-цель
*   цена товара и выручка в датасете отсутствуют, поэтому модель не оптимизирует revenue напрямую
*   для production-решения нужна онлайн-валидация через A/B-тест
