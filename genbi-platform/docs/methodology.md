# Методология проекта MWS Data Compass GenBI

## Контекст проекта

Проект **MWS Data Compass GenBI** подготовлен как добровольная инициатива в рамках программы **«Идея на триллион»**, где сотрудники предлагают изменения и улучшения для компании на добровольной основе.

Проект описывает информационную систему для получения аналитики через естественный язык на базе существующей платформы MWS Data. Решение не предполагает замену текущей инфраструктуры. Основная идея состоит в добавлении GenBI-слоя, который связывает существующие компоненты в единый пользовательский сценарий.

## 1. Назначение и цели

### 1.1. Наименование системы

Рабочее наименование: **Информационная система «MWS Data Compass GenBI»**.

Далее: ИС, система, платформа или GenBI-слой.

### 1.2. Назначение системы

ИС предназначена для обеспечения доступа к аналитике через естественный язык (**Natural Language Query, NLQ**) на платформе MWS Data Compass.

Система позволяет бизнес-пользователям получать аналитические результаты без написания SQL и без обязательного участия аналитика в типовых сценариях.

Результатом работы системы могут быть:

*   SQL-запрос
*   табличный результат
*   визуализация
*   BI-дашборд
*   ссылка на сохраненный артефакт в Data Compass

### 1.3. Цели создания системы

| Категория | Цель | Ожидаемый эффект |
|---|---|---|
| Бизнес | Сократить time-to-insight | С нескольких дней до менее 5 минут на типовом запросе |
| Бизнес | Снизить нагрузку на аналитиков | Автоматизация рутинных ad-hoc задач |
| Технологии | Создать единую точку входа в аналитику | Пользователь работает через текстовый запрос |
| Технологии | Сохранить результаты как BI-артефакты | Датасет, график и дашборд создаются в Data Compass |
| Организация | Стандартизировать метрики | Единое определение метрик через Semantic Layer |
| Организация | Масштабировать подход на команды | Возможность применения в разных аналитических доменах |

### 1.4. Ключевой принцип работы

Основной принцип системы — **Self-Service Analytics**.

Бизнес-пользователь не должен знать:

*   SQL
*   структуру таблиц
*   физическую модель данных
*   внутренние правила BI-инструмента

Система автоматически:

*   интерпретирует текстовый запрос
*   находит нужные метрики
*   подбирает таблицы и dimensions
*   генерирует SQL
*   проверяет SQL перед выполнением
*   выполняет запрос с учетом прав пользователя
*   строит визуализацию
*   возвращает результат с источником и датой обновления данных

При этом аналитики остаются владельцами методологии: они развивают Semantic Layer, контролируют качество метрик и занимаются сложным анализом.

## 2. Бизнес-проблематика

В текущем процессе бизнес-пользователь зависит от аналитика почти в каждом запросе к данным. Даже простой вопрос, например расчет продаж по региону за месяц, требует постановки задачи, ожидания очереди, написания SQL и сборки визуализации.

Ключевые проблемы:

*   бизнес не владеет SQL и структурой данных
*   аналитик становится обязательным звеном между бизнесом и данными
*   типовые задачи занимают значительную часть времени аналитических команд
*   time-to-insight измеряется днями, хотя бизнесу нужен ответ за минуты
*   BI, metadata, SQL-движок и AI существуют как отдельные компоненты
*   нет orchestration-слоя, который связывает их в единый runtime-сценарий

Текущий поток:

```text
Source Systems → Storage Layer → Data Query → Data Compass → Analyst → Business User
```

Целевой поток:

```text
Business User → AI Agent → Semantic Layer → SQL Validator → Data Query → Data Compass → Analytical Result
```

Главный вывод:

```text
Разрыв не технологический, а архитектурный: компоненты есть, но связки между ними нет.
```

## 3. Use Cases

### UC_1. Получение аналитики по текстовому запросу

| Поле | Описание |
|---|---|
| Идентификатор | UC_1 |
| Название | Получение аналитики по текстовому запросу |
| Основное действующее лицо | Бизнес-пользователь |
| Цель | Получить аналитический результат без SQL и обращения к аналитику |
| Приоритет | Высокий |
| Частота использования | Очень высокая |

#### Описание

Пользователь формулирует аналитический вопрос на естественном языке. AI Agent интерпретирует запрос, находит подходящую метрику в Semantic Layer, генерирует SQL, выполняет запрос через Data Query и возвращает результат с визуализацией.

#### Предварительные условия

*   пользователь авторизован
*   Data Cat содержит релевантные метаданные
*   Semantic Layer содержит метрики выбранного домена
*   пользователь имеет доступ к нужным данным

#### Основной поток

1. Пользователь вводит текстовый запрос.
2. AI Agent определяет тип запроса.
3. Agent выполняет поиск по Semantic Layer и Data Cat.
4. Найдена метрика с достаточной уверенностью.
5. Agent проверяет доступные dimensions.
6. Генерируется SQL.
7. SQL Validator проверяет запрос.
8. Auth / Permission Gateway проверяет права.
9. SQL выполняется через Data Query с impersonation.
10. Agent выбирает тип визуализации.
11. Пользователь получает результат с Data Freshness.

#### Альтернативные потоки

*   Если уверенность средняя, Agent задает уточняющий вопрос.
*   Если уверенность низкая, Agent переходит в ad-hoc text-to-SQL режим с предупреждением.
*   Если запрошенный dimension недоступен, система показывает доступные варианты.

#### Исключения

*   SQL Validator не смог исправить запрос за 3 попытки.
*   У пользователя нет доступа к данным.
*   Data Query недоступен.

### UC_2. Публикация результата как BI-артефакта

| Поле | Описание |
|---|---|
| Идентификатор | UC_2 |
| Название | Публикация результата как BI-артефакта |
| Основное действующее лицо | Бизнес-пользователь |
| Цель | Сохранить результат как дашборд в Data Compass |
| Приоритет | Высокий |

#### Описание

После получения аналитического результата пользователь запрашивает сохранение в виде дашборда. BI Publisher создает виртуальный датасет, чарт и дашборд в Data Compass. Data Cat получает новые артефакты и строит lineage до источника.

#### Основной поток

1. Пользователь запрашивает сохранение результата.
2. Agent формирует VizSpec и DashboardSpec.
3. BI Publisher валидирует параметры визуализации.
4. Создается виртуальный датасет на SQL.
5. Создается чарт.
6. Создается дашборд с учетом прав и RLS.
7. Data Cat получает новые объекты.
8. Строится lineage.
9. Пользователь получает ссылку на дашборд.

### UC_3. Уточнение запроса в диалоговом режиме

| Поле | Описание |
|---|---|
| Идентификатор | UC_3 |
| Название | Уточнение запроса в диалоговом режиме |
| Основное действующее лицо | Бизнес-пользователь |
| Цель | Устранить неоднозначность запроса |
| Приоритет | Высокий |

#### Описание

Если запрос неоднозначен, AI Agent задает уточняющий вопрос. Пользователь выбирает вариант или вводит свободный ответ. Agent обновляет Conversation State и продолжает выполнение.

#### Основной поток

1. AI Agent обнаруживает неоднозначность.
2. Agent формулирует уточняющий вопрос.
3. Пользователь выбирает вариант.
4. Agent обновляет Conversation State.
5. Выполнение продолжается по UC_1.

### UC_4. Управление Semantic Layer аналитиком

| Поле | Описание |
|---|---|
| Идентификатор | UC_4 |
| Название | Управление Semantic Layer аналитиком |
| Основное действующее лицо | Аналитик |
| Цель | Добавлять, обновлять и версионировать метрики |
| Приоритет | Средний |

#### Описание

Аналитик добавляет или обновляет метрику в Semantic Layer. Изменения проходят через Git и CI/CD-тесты.

#### Основной поток

1. Аналитик изменяет YAML-конфиг метрики.
2. CI/CD запускает тесты.
3. При успешных тестах изменение проходит merge.
4. Agent подхватывает новую версию Semantic Layer.
5. При breaking change зависимые метрики переводятся в draft.

## 5. Архитектура системы

### 5.1. Общая архитектурная концепция

Система проектируется как надстройка над существующей платформой MWS Data. Подход основан на принципе **orchestrator-based pipeline**.

Новый AI Agent / GenBI Layer принимает запрос и управляет вызовами сервисов через явные инструменты.

### 5.2. Компоненты AS-IS

| Компонент | Роль | Ограничение |
|---|---|---|
| MWS Data Cat | Каталог метаданных, glossary, lineage | Не используется как runtime Semantic Layer |
| MWS Data Query | Выполнение SQL через Trino | Требует знания SQL |
| MWS Data Compass | BI, дашборды | Требует ручной сборки |
| MWS GPT + RAG | LLM-платформа | Не встроена в BI-сценарий |
| Storage Layer | Хранение данных | Требует знания структуры |

### 5.3. Компоненты TO-BE

| Компонент | Описание | Техническая опора |
|---|---|---|
| AI Agent / GenBI Layer | Оркестратор NLQ-сценария | MWS GPT + RAG |
| Semantic Layer | Метрики, dimensions, SQL-шаблоны, synonyms | Data Cat |
| SQL Validator | Проверка SQL, schema check, EXPLAIN dry-run | Trino |
| Auth / Permission Gateway | Guardrails, impersonation, RLS | Data Query / Rules Generator |
| BI Publisher | Создание датасета, чарта и дашборда | Data Compass API |
| Conversation State | Контекст диалога | Redis |

### 5.4. TO-BE поток

```text
Business User
    ↓
AI Agent / GenBI Layer
    ↓
Semantic Layer + Data Cat
    ↓
SQL Validator
    ↓
Auth / Permission Gateway
    ↓
Data Query
    ↓
BI Publisher / Data Compass
    ↓
Analytical Result
```

*   vector search

## 14. Итог

MWS Data Compass GenBI — это добровольная проектная инициатива в рамках программы «Идея на триллион», направленная на развитие self-service аналитики внутри компании.

Проект показывает, как существующие компоненты MWS Data можно связать в единый GenBI-сценарий:

```text
естественный язык → метрика → SQL → проверка → выполнение → визуализация → BI-артефакт
```

Ключевая ценность проекта — сокращение time-to-insight, снижение рутинной нагрузки на аналитиков и повышение доступности данных для бизнеса без замены текущей платформы.
