# MWS Data Compass GenBI

## Описание проекта

MWS Data Compass GenBI — добровольный проект в рамках внутренней программы ПАО МТС «Идея на триллион».

В рамках программы сотрудники предлагают инициативы по улучшению процессов, продуктов и внутренних систем компании на добровольной основе. Данный проект описывает концепцию GenBI-слоя для платформы MWS Data Compass, который позволяет получать аналитику через запросы на естественном языке.

Проект не предполагает замену существующей инфраструктуры. Основная идея состоит в том, чтобы связать уже существующие компоненты MWS Data через новый orchestration-слой и создать единый пользовательский сценарий: от вопроса бизнес-пользователя до готового аналитического результата.

---

## Проблема

В текущем процессе бизнес-пользователь зависит от аналитика почти в каждом запросе к данным.

Основные ограничения:

*   бизнес-пользователь не знает SQL и структуру таблиц
*   аналитик остается обязательным звеном между бизнесом и данными
*   типовые ad-hoc задачи занимают значительную часть времени аналитических команд
*   time-to-insight измеряется днями, хотя бизнесу нужен ответ за минуты
*   BI, SQL, metadata и AI-компоненты существуют отдельно и не связаны в единый пользовательский сценарий

Текущий поток:

```text
Source Systems → Storage Layer → Data Query → Data Compass → Analyst → Business User
```

Ключевой разрыв:

```text
Компоненты есть, связки нет
```

## Цель проекта

Создать GenBI-слой поверх MWS Data Compass, который позволит:

*   принимать аналитические запросы на естественном языке
*   сопоставлять бизнес-термины с метриками и таблицами
*   генерировать SQL-запросы
*   проверять SQL перед выполнением
*   выполнять запросы с учетом прав пользователя
*   строить визуализации и BI-артефакты
*   снижать нагрузку на аналитиков за счет self-service аналитики

---

## Целевой пользовательский сценарий

```text
Business User → AI Agent → Semantic Layer → SQL Validator → Data Query → Data Compass → Analytical Result
```

Пример запроса:

```text
Покажи выручку по регионам за март 2026
```

Система:

*   определяет аналитический intent
*   находит метрику «выручка» в Semantic Layer
*   подбирает нужные таблицы через Data Cat
*   генерирует SQL
*   проверяет запрос через SQL Validator
*   выполняет SQL через Data Query
*   строит визуализацию в Data Compass
*   возвращает ссылку на результат с источником и датой обновления данных

## Ключевые компоненты

### 1. AI Agent / GenBI Layer

Оркестратор пользовательского сценария. Принимает NLQ-запрос, определяет intent, управляет вызовом Semantic Layer, SQL Validator, Data Query и Data Compass.

### 2. Semantic Layer

Слой бизнес-метрик и терминов. Сопоставляет пользовательские формулировки с физической моделью данных.

Пример:

```yaml
metrics:
  revenue:
    sql: SUM(orders.amount)
    filters: "status = 'paid'"
    source: orders

synonyms:
  выручка: revenue
  продажи: revenue
```

### 3. SQL Validator

Компонент проверки SQL перед выполнением.

Проверяет:

*   синтаксис SQL
*   существование таблиц и колонок
*   соответствие схеме
*   безопасность запроса
*   стоимость выполнения

При ошибке запускается self-correction loop до 3 попыток.

### 4. Auth / Permission Gateway

Контролирует права пользователя.

Основные функции:

*   RBAC
*   RLS
*   impersonation
*   SQL guardrails
*   запрет DDL/DML
*   выполнение только SELECT-запросов

### 5. BI Publisher

Создает BI-артефакты в Data Compass:

*   virtual dataset
*   chart
*   dashboard
*   lineage до источника данных

### 6. Conversation State

Хранит контекст диалога пользователя:

*   выбранные метрики
*   уточнения
*   предыдущие SQL-запросы
*   состояние сессии

## Use Cases

### UC_1. Получение аналитики по текстовому запросу

Пользователь вводит вопрос на естественном языке. Система интерпретирует запрос, находит нужную метрику, генерирует SQL, выполняет запрос и возвращает результат.

### UC_2. Публикация результата как BI-артефакта

Пользователь сохраняет результат анализа как дашборд в Data Compass.

### UC_3. Уточнение запроса в диалоговом режиме

Если запрос неоднозначен, агент задает уточняющий вопрос и продолжает выполнение после ответа пользователя.

### UC_4. Управление Semantic Layer

Аналитик обновляет метрики, бизнес-термины и SQL-шаблоны через версионируемый Semantic Layer.

## Roadmap

### Этап 1. Text-to-SQL, 0–3 месяца

Цель: запустить базовый сценарий получения аналитики через текстовый запрос.

Компоненты:

*   AI Agent
*   базовый Semantic Layer
*   SQL Validator
*   Conversation State
*   Data Cat integration
*   Data Query integration

### Этап 2. BI Publisher + Semantic Layer, 3–6 месяцев

Цель: сохранять результаты агента как BI-артефакты.

Компоненты:

*   BI Publisher
*   расширенный Semantic Layer
*   auto-ingestion в Data Cat
*   lineage

### Этап 3. Полноценный GenBI, 6–12 месяцев

Цель: типовые аналитические запросы закрываются без участия аналитика.

Компоненты:

*   feedback loop
*   observability
*   async execution
*   caching
*   масштабирование

### Этап 4. Decision Support, 12+ месяцев

Цель: перейти от реактивной аналитики к поддержке принятия решений.

Компоненты:

*   proactive insights
*   recommendations
*   forecasting
*   multi-agent architecture
